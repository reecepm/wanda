# Frontend Cleanup Goal

This document is the operating contract for the Wanda frontend cleanup loop. It should be read before each cleanup iteration, agent assignment, or architecture enforcement change.

## Goal

Bring Wanda's frontend to a consistently enforceable renderer architecture where TanStack Router route modules are the screen definitions, feature folders own reusable product behavior, server state flows through oRPC and TanStack Query, UI state stays deliberately local or feature-scoped, and code quality is guarded by automated checks wherever practical and by this contract everywhere else.

The cleanup is not complete after the first pass of fixes. It is complete only when the same analysis can be re-run recursively and finds no material violations of these standards, except documented exceptions with ownership and justification.

## Target Architecture

Frontend code should converge on this shape:

- `src/routes/**` owns route modules and top-level screen composition.
- `src/routeTree.gen.ts` is generated by TanStack Router and must not be edited manually.
- `src/pages/**` is deprecated. Screen definitions should live directly in `src/routes/**`.
- `src/features/<feature>/components/**` owns reusable UI for that feature.
- `src/features/<feature>/hooks/**` owns feature data hooks, subscriptions, and renderer-side orchestration.
- `src/features/<feature>/store/**` owns feature-local UI, editor, selection, layout, draft, or preference state.
- `src/features/<feature>/utils/**` owns pure feature transforms, formatters, parsers, and classification helpers.
- `src/features/<feature>/index.ts` is the public feature API for cross-feature imports.
- `src/ui/**` owns shared design-system primitives only.
- `src/layout/**` owns app shell, navigation, top bars, and global app chrome.
- `src/shared/**` owns renderer infrastructure such as oRPC clients, query helpers, transport bridges, persistence helpers, and cross-feature utilities.
- `src/stores/**` owns small app-wide UI/preference stores only.
- `src/tray/**` owns the tray renderer surface.
- `shared/contracts/**` owns all client/server boundary schemas and types.

Route modules are the reference direction, not dumping grounds. A route may own URL params, search validation, loaders/preload, pending/error UI, and screen composition. Substantial behavior, reusable panels, data hooks, and mutation workflows should move into the relevant feature folder.

## Hard Rules

These should be automated with Biome, ESLint, custom scripts, typecheck, Storybook checks, or tests wherever practical.

- New route screens must be defined directly in `src/routes/**`, not in `src/pages/**`.
- `src/pages/**` must not grow. Existing pages should be inlined into matching route modules and deleted during cleanup.
- `src/routeTree.gen.ts` must not be edited manually.
- Route modules must import feature code through `@/features/<feature>` barrels for cross-feature dependencies.
- Cross-feature imports must go through `src/features/<feature>/index.ts`. Deep imports are allowed only inside the same feature.
- Renderer production code must not import from `electron/**`, except approved preload/type boundary files.
- Renderer code must not import backend implementation types directly. Types crossing the client/server boundary must flow through `shared/contracts/**` or the existing renderer compatibility barrel.
- Server state belongs in TanStack Query through `orpc`, `orpcUtils`, or feature query helpers. Do not mirror server state in Zustand without a documented reason.
- Zustand stores are for UI/editor/selection/layout/draft/preference state, not canonical backend data.
- Feature components should not call raw `window.wanda.*` unless they are the owning transport, native capability, or hardware integration feature.
- Mutations that change server state must invalidate or update the exact affected TanStack Query keys.
- New production frontend code must not introduce casual `any`, `as unknown as`, `@ts-ignore`, `@ts-expect-error`, or broad double casts without an exception entry.
- React components and hooks must stay render-pure. Side effects, subscriptions, random IDs, timestamps, DOM mutation, and global mutation belong in event handlers, effects, lazy initializers, stores, or adapters as appropriate.
- Hook dependency correctness must be enforced by `eslint-plugin-react-hooks` in the normal lint path.
- Shared UI should use `src/ui/**` primitives and theme tokens rather than ad hoc controls.
- Touched interactive UI needs keyboard/focus/accessibility consideration through Storybook, browser/e2e coverage, focused tests, or a written reason why existing coverage is enough.
- Changed frontend code must pass `bun run typecheck`.

## Soft Rules

These require engineering judgment and should be reviewed during each iteration.

- Keep route modules as composition boundaries. A route doing complex query orchestration, subscriptions, mutation workflows, or large local component trees should usually delegate to feature hooks/components.
- Prefer feature hooks such as `useWorkenvList`, `usePodData`, `useGitStatus`, or a new feature-specific hook over repeated inline `useQuery` blocks.
- Prefer route loaders or route-level preload for route-critical async work when the route knows enough to coordinate it early.
- Prefer `orpcUtils.*.queryOptions`, `orpcUtils.*.queryKey`, and feature query-key helpers over hand-written query keys.
- Prefer invalidate-on-success for simple mutations and explicit cache updates only when they reduce visible churn or preserve interaction state.
- Prefer pure utility tests for filtering, sorting, parsing, workflow transforms, layout transforms, query-key helpers, and store reducers.
- Prefer Storybook stories for shared primitives, dialogs, dense panels, menus, and complicated feature surfaces.
- Keep app-wide stores small. If state only matters to one feature, keep it in that feature.
- Keep feature barrels intentional. Export reusable public API, not every implementation detail.
- Delete dead components, hooks, stores, and exports once proven unused. Do not preserve stale exports without an owner and removal condition.
- Avoid broad visual rewrites during architecture cleanup unless the assigned feature explicitly requires it.

## Automation Goals

Enforcement should be tightened in stages so each iteration can make measurable progress.

- Add a frontend lint bundle, `bun run lint:frontend`, parallel to `bun run lint:backend`.
- Include `bun run lint:frontend` in the main `bun run lint` path.
- Include ESLint in the normal lint path so `eslint-plugin-react-hooks` enforces hook and render correctness.
- Keep `bun run lint:barrels` as hard enforcement for cross-feature barrel imports.
- Add Biome `noRestrictedImports` coverage for renderer/backend boundary violations and new `src/pages/**` usage where practical.
- Add a custom frontend architecture lint for rules Biome cannot express precisely, including raw `window.wanda.*` and direct `orpc` use outside approved files.
- Add a frontend unsafe-type budget for production files, with documented exceptions and removal conditions.
- Add a dead-code/export audit for `src/features/**`, `src/ui/**`, `src/shared/**`, `src/layout/**`, and `src/stores/**`.
- Add or tighten Storybook accessibility checks for shared UI and important feature surfaces.
- Track oversized route modules and feature files as an audit signal, not a standalone failure condition.

## Recursive Cleanup Loop

1. Read this document and confirm the current cleanup scope.
2. Run the existing checks and relevant discovery queries to establish a violation baseline.
3. Assign bounded work by route, feature, or rule category. Agents must not overlap write ownership unless explicitly coordinated.
4. Make behavior-preserving fixes unless a behavior bug is explicitly identified.
5. Move screen definitions from `src/pages/**` into matching `src/routes/**` modules.
6. Extract reusable or complex route-local code into feature components/hooks/stores/utils.
7. Run targeted tests, Storybook checks, or e2e tests for touched areas.
8. Run `bun run typecheck`.
9. Re-run architecture lint and the original discovery queries.
10. Update this contract or the automation if a new recurring violation category appears.
11. Repeat until the re-analysis produces no material violations.
12. Do a final confidence pass: typecheck, frontend lint bundle, targeted tests, Storybook/a11y coverage for touched UI, and manual review of exceptions/dead-code candidates.

## Definition Of Done

The cleanup is done when:

- Frontend architecture checks pass.
- `bun run typecheck` passes.
- `bun run lint` includes the frontend lint bundle and passes.
- Screen definitions live in `src/routes/**`; `src/pages/**` is empty or deleted.
- Route modules are composition boundaries, not large behavior containers.
- Cross-feature imports go through feature barrels.
- Renderer production code has no backend/Electron implementation imports outside documented boundaries.
- Server state is managed through TanStack Query/oRPC, not duplicated into Zustand without documented justification.
- No new unsafe typing escapes exist in production frontend code, beyond documented exceptions with owner, justification, and removal condition.
- Shared UI and substantial touched feature surfaces have focused tests, Storybook stories, browser/e2e coverage, or documented coverage rationale.
- Dead frontend code candidates have been deleted or documented as intentionally retained.
- A final recursive scan produces no new categories of violations.
